# flower

> flow + er —— 一个面向开发者的流程引擎重构项目。

`flower` 是你之前项目 [rlang](https://github.com/RaindropSaber/rlang) 的下一代版本：
- 从「实验验证」升级到「可落地、可扩展、可维护」；
- 从「仅能跑通流程」升级到「支持建模、执行、可观测、治理」的完整工程。

## 愿景

让流程定义像写代码一样可组合，让流程执行像基础设施一样稳定可靠。

## 核心目标

- **声明式建模**：使用简洁 DSL / JSON / YAML 描述流程。
- **可靠执行**：支持状态持久化、失败重试、补偿逻辑。
- **可观测性**：提供执行日志、节点耗时、链路追踪。
- **可扩展性**：插件化 Task/Node 扩展机制。
- **工程化能力**：版本化、回放、灰度发布、权限控制（后续阶段）。

## MVP 范围（第一阶段）

1. **流程定义**
   - DAG 流程（起点、任务节点、分支、结束）
   - 基础条件分支（if/else）
2. **执行引擎**
   - 单实例串行执行
   - 失败重试（固定次数）
   - 执行状态落盘（SQLite 或文件）
3. **开发者接口**
   - CLI：`validate` / `run` / `inspect`
   - 最小 SDK：注册任务处理器
4. **可观测性（基础）**
   - 结构化日志
   - execution id 级别查询

## 分层架构（建议）

- **Model 层**：流程定义、节点定义、上下文数据模型
- **Compiler 层**：定义校验、拓扑排序、可执行计划生成
- **Runtime 层**：调度、执行、重试、状态机
- **Persistence 层**：流程实例状态、事件日志、快照
- **Interface 层**：CLI / API / SDK

## 里程碑

- **M1: 跑通最小流程**（定义 + 执行 + 日志）
- **M2: 可靠性增强**（重试 + 状态恢复）
- **M3: 扩展能力**（插件节点 + 子流程）
- **M4: 可观测与治理**（监控指标 + 权限 + 发布流程）

## 命名说明

`flower = flow + er`

既表达“流程（flow）”核心，也表达“执行者（-er）”角色。
