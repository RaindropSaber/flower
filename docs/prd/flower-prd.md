## 1. 背景与问题定义

在 AI 能力增强和自然语言交互普及的背景下，用户希望通过“对话”完成复杂电脑操作。

但当前通用 AI Agent 直接操控桌面存在明显问题：

1. **不稳定**：同一指令在不同环境下执行结果不一致。
2. **不可控**：执行步骤不透明，难以设置精细化边界。
3. **高风险**：误操作可能导致文件删除、误发消息、错误提交等。
4. **不可复用**：一次对话难以沉淀为可复用、可维护的自动化资产。

### 1.1 机会点

构建一个“桌面端流程引擎”，让 AI 负责理解意图与生成流程，执行交给可控的工程化运行时。

---

## 2. 产品愿景与定位

### 2.1 愿景

让普通用户通过自然语言构建自动化流程，并以工程级可靠性在本地/桌面端安全执行。

### 2.2 产品定位（一句话）

**Flower 是一个 AI 时代的桌面流程引擎：用自然语言搭建流程，用工程化机制保证执行安全、稳定、可审计。**

### 2.3 命名

`Flower = flow + er`

- `flow`：流程
- `er`：执行者（engine / runner）

---

## 3. 目标与非目标

### 3.1 产品目标（MVP）

1. 用户可通过自然语言生成可编辑的流程草案（DAG）。
2. 用户可在发布前进行人工确认与风险预览。
3. 引擎可稳定执行流程，支持失败重试与状态恢复。
4. 全流程可审计：可查看每次执行的节点日志与结果。

### 3.2 非目标（当前阶段不做）

1. 暂不做多端云协同（仅桌面单机优先）。
2. 暂不做复杂多租户权限系统。
3. 暂不做完整低代码可视化设计器（先保证核心链路）。
4. 暂不覆盖所有系统自动化场景（先做高频节点）。

---

## 4. 目标用户与核心场景

### 4.1 目标用户

1. **个人效率用户**：希望自动化重复电脑操作（文件整理、日报生成、批处理）。
2. **轻技术用户**：可写提示词但不想写脚本。
3. **开发者/自动化爱好者**：需要可扩展节点能力与可复用流程模板。

### 4.2 核心场景（MVP）

1. 文件批处理：按规则重命名、归档、格式转换。
2. 信息汇总：从网页/本地文档采集信息并生成报告。
3. 通知触发：流程执行完成后发送消息/邮件。

---

## 5. 核心能力需求

### 5.1 自然语言建流（NL2Flow）

- 输入：自然语言目标（例：“每天 9 点汇总下载文件夹中的发票并生成表格”）
- 输出：结构化流程定义（节点、边、条件、输入输出）
- 能力要求：
  - 生成流程草案并解释关键假设
  - 高亮不确定字段并要求用户补充
  - 支持二次对话迭代流程

### 5.2 流程编辑与发布

- 流程可视化/结构化查看
- 节点参数可编辑
- 发布前校验：依赖、参数、权限、风险级别

### 5.3 执行运行时（Runtime）

- 串行执行（MVP）
- 失败重试（固定次数 + 间隔）
- 执行状态持久化（可恢复）
- Execution ID 全链路追踪

### 5.4 安全控制（Safety Guard）

- 节点能力分级（读/写/执行/外发）
- 高风险节点人工确认（可配置）
- 目标路径/域名白名单
- 敏感操作前置预览（删除、覆盖、发送）

### 5.5 可观测与审计

- 结构化日志（节点开始/成功/失败）
- 节点耗时、重试次数、错误摘要
- 执行回放（最小版本先从日志回放）

---

## 6. 信息架构与模块划分

1. **Flow Definition**：流程定义模型（DAG、Node、Condition）
2. **Compiler/Validator**：语义校验、拓扑检查、执行计划生成
3. **Runtime Engine**：调度、执行、重试、状态机
4. **Node Registry/Plugin**：节点注册、插件扩展
5. **Safety Guard**：权限与风险策略
6. **Persistence**：实例状态、事件日志、快照
7. **Interface**：CLI/API/SDK（MVP 先 CLI + SDK）

---

## 7. 关键用户流程（MVP）

1. 用户输入自然语言任务
2. 系统生成流程草案 + 风险提示
3. 用户调整节点参数并确认发布
4. 系统执行流程并实时记录日志
5. 用户查看执行结果与失败节点详情
6. 用户可再次运行或复制为模板

---

## 8. 里程碑规划

### M1（基础可跑）

- 流程定义模型
- DAG 校验
- 串行执行 + 结构化日志

### M2（可靠执行）

- 重试机制
- 状态持久化与失败恢复
- execution 检索能力

### M3（自然语言与扩展）

- NL2Flow 草案生成
- 节点注册机制（插件化）
- 模板化流程

### M4（安全治理）

- 风险分级与人工确认网关
- 白名单策略
- 基础审计报表

---

## 9. 成功指标（Success Metrics）

### 9.1 产品指标

1. 流程首次构建成功率（用户首次发布成功）
2. 流程执行成功率（7 日窗口）
3. 人工确认拦截率（高风险动作）
4. 流程复用率（被重复执行次数）

### 9.2 体验指标

1. 从自然语言到可发布流程的中位时长
2. 平均调试轮次（发布前）
3. 失败定位时间（MTTD）

---

## 10. 风险与应对

1. **LLM 生成不稳定**
   - 应对：流程 DSL 强约束 + 校验器兜底 + 人工确认
2. **桌面环境差异导致节点失败**
   - 应对：能力探测 + 节点前置检查 + 可恢复重试
3. **安全事故风险**
   - 应对：默认最小权限 + 风险网关 + 审计追踪

---

## 11. 首批节点清单（建议）

1. 文件读取
2. 文件写入
3. 文件移动/重命名
4. CSV/Excel 读取
5. HTTP 请求
6. 浏览器打开页面
7. 文本提取（规则/LLM）
8. 条件分支
9. 通知发送（Webhook/邮件）
10. 人工确认节点

---

## 12. 发布标准（MVP Exit Criteria）

满足以下条件可认为 MVP 可发布：

1. 可完成“自然语言 -> 流程草案 -> 人工确认 -> 执行 -> 审计”的闭环。
2. 具备失败重试与重启恢复能力。
3. 高风险操作默认需确认。
4. 至少 5 个高频节点在真实场景稳定可用。
